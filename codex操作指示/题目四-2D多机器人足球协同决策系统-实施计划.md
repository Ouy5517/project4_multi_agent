# Booster T1 3D 多机器人足球协同决策系统 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 保留已经通过验收的 2D 协同决策系统，把当前圆柱机器人驱动的 MuJoCo 2.5D 展示升级为使用官方 Booster T1 模型、MuJoCo 物理、关节控制、足球接触和实时 3D 画面的多机器人足球仿真。

**Architecture:** 继续复用 `WorldState -> TeamCoordinator/DecisionFSM -> RobotAction` 策略主干，在旁边新增独立的 `simulation3d/` 后端。MuJoCo 成为 3D 模式中机器人、足球、碰撞和时间的唯一真源；`WorldState` 从 MuJoCo 状态读取，`RobotAction` 经 T1 官方模式语义、全向移动控制器和踢球控制器作用到关节，渲染器只显示同一份 MuJoCo 状态。

**Tech Stack:** WSL2 Ubuntu 22.04、Python 3.10、MuJoCo 3.10、NumPy、PyTorch CPU、pytest、官方标准版 T1 23-DoF 模型、官方 `BoosterRobotics/booster_gym` T1 locomotion 资源、可选 ROS2 Humble/Booster Robotics SDK。

**Progress snapshot:** 2026-07-14，当前分支 `feature/2d-soccer-system`，HEAD `8deea36`。

## 0. 需求纠正与术语冻结

### 0.1 原计划中的错误

原计划把“2D 仿真”主要解释为可无窗口运行的数值仿真，并写成“主线不依赖 MuJoCo、3D 或机器人模型”。这不符合项目展示目标。

本项目所说的“2D 仿真”应当是：

- 有完整足球场、边界、球门和足球；
- 有可辨认的双方机器人图形、朝向、编号、角色或状态；
- 球和机器人根据决策结果实时运动；
- 能打开窗口观看，也能录制或回放；
- 无窗口模式仅用于自动测试、性能测试和批量验收，不能代替可视化成果。

### 0.2 当前升级后的最终目标

用户现已把最终目标从可视化 2D 仿真改为 **Booster T1 3D 仿真**。三种模式必须严格区分：

| 名称 | 定义 | 当前状态 | 能否作为最终 3D 成果 |
| --- | --- | --- | --- |
| 2D 基线 | 俯视足球场、机器人图形和球实时运动，可输出 PNG/MP4 | 已完成 | 否，只保留为快速策略测试基线 |
| 2.5D 预览 | MuJoCo 中用圆柱体表示机器人，位置由 2D 仿真每帧写入 freejoint | 已完成 | 否，不是 Booster T1，也不是 3D 动力学闭环 |
| Booster T1 3D | 官方 T1 关节模型、多机器人、足球场、MuJoCo 物理、关节控制、接触踢球和实时相机 | 未完成 | 是，本计划的最终交付目标 |

### 0.3 “Booster T1 3D 已完成”的硬门槛

只有同时满足以下条件，才允许在 README、报告或答辩中写“已完成 Booster T1 3D 仿真”：

- 场景中的我方机器人加载自官方 T1 XML/mesh，而不是圆柱、胶囊或贴图替代物；
- 至少两台 T1 同时存在，模型名称、关节和控制通道互不冲突；
- MuJoCo 的 `qpos/qvel/contact` 是 3D 模式的真实状态源；
- 正常运行时不得每帧直接覆盖机器人根节点位置来伪造行走；
- 移动和转向由关节控制器或经验证的 locomotion policy 产生；
- 最终传球验收中的球速变化来自脚与球的 MuJoCo 接触；直接修改球速度只能作为明确标记的 `mock_kick` 调试模式；
- 窗口中能看到足球场、球门、足球和实时运动的 Booster T1；
- 无窗口自动测试与实时窗口使用同一个 3D 世界和控制后端。

| `https://github.com/BoosterRobotics/booster_gym` | `da396a06d6eed99e2de72d7749c48ee8748950f9` | T1 MuJoCo locomotion 参考、观察量、PD 参数和 `deploy/models/T1.pt` |
| `https://github.com/BoosterRobotics/booster_robotics_sdk` | 仅实机/平台桥接时锁定 | 官方高层 `Move`、`Shoot` 等接口参考 |
| `https://github.com/BoosterRobotics/robocup_demo` | `fa4f31ccb4428a9e60e90b4f54eb80499caeb0ca` | ROS2 足球决策与虚拟机器人联调参考 |

已核实 `booster_assets` 同时提供 T1 23 DoF、29 DoF 和 locomotion 变体。本项目最终场景固定使用标准版 `T1_23dof.xml`：头部 2 DoF、双臂 4 DoF x 2、腰部 1 DoF、双腿 6 DoF x 2。locomotion policy 只控制 12 个腿部关节，其余 11 个上身关节由姿态保持控制器稳定在配置姿态。

不选 `T1_29dof`（7-DoF 手臂版本），因为官方 T1 说明书明确注明 7-DoF 版本不支持 Soccer Agent。`T1_locomotion.xml` 只用于对照官方 locomotion 关节、sensor 和 PD 契约，不作为最终视频中的机器人模型。

官方说明书的开发目录包含 Webots、Isaac、SDK 和 ROS2 路线；本计划选择 MuJoCo 是因为课程足球平台要求和官方 `booster_assets`/`booster_gym` 已提供可验证的 MuJoCo T1 资源。文档中不得把该技术选型误写成 T1 说明书唯一指定方案。

### 2.2 T1 机型特性到系统设计的映射

| 官方 T1 特性 | 本项目对应设计 | 自动验收 |
| --- | --- | --- |
| 标准版全身 23 DoF | 场景始终加载 `T1_23dof.xml`，上身 11 关节保持姿态，下肢 12 关节执行 locomotion | body/joint/actuator 名称与数量契约测试 |
| 头部 2 DoF、带深度相机 | 保留 head yaw/pitch；每台 T1 建立第一视角 RGB/depth 相机，基础策略仍读取真值 WorldState | 相机帧非空、depth 有限值测试 |
| 前向、后向、侧向行走 | `BodyVelocityCommand(vx, vy, yaw_rate)` 支持正负 `vx/vy` | 前进、后退、左移、右移位移测试 |
| 旋转及复合行走 | path follower 可同时输出平移与 yaw rate | 90 度转向与斜向复合运动测试 |
| WALK 模式有更强抗扰能力 | 足球运行期间保持 WALK 的主动平衡；PREP 只用于模式过渡和静态站立 | 固定侧向推力后的恢复测试 |
| DAMP/PREP/WALK/CUSTOM/PROTECT 模式 | 3D 后端实现同名模式机和官方允许的转换路径 | 非法转换拒绝与失败码测试 |
| Default Agent 含爬起等预置动作 | 本地 3D 第一阶段跌倒进入 PROTECT；自动爬起作为独立增强，不用 reset 冒充爬起 | 跌倒检测和 PROTECT 测试 |
| Soccer Agent 支持盯球、追球 | 现有 CHASE 策略映射为 WALK 全向速度；头部可选跟踪球 yaw/pitch | 追球距离单调下降测试 |
| 失控自动保护、硬急停 | NaN、跌倒、超关节范围进入 PROTECT；仿真提供 `--e-stop-file` 软件急停并立即 DAMP；实机仍必须使用物理硬急停 | 故障注入和软件急停延迟测试 |

官方资料只定性说明“承受一定冲击扰动”，没有给出公开量化阈值。本项目把“躯干承受 40 N 侧向力 0.1 秒后，2 秒内恢复到躯干高度不低于 0.60 m 且 roll/pitch 绝对值不超过 20 deg”定义为课程仿真基准；该数值是本项目验收指标，不宣称为 Booster 官方性能参数。

### 2.3 资源管理规则

- 官方 mesh、模型和 policy 放在 `external/`，由脚本按固定提交下载；不直接复制到课程源码目录。

- 如果官方资源许可证不允许再分发，演示机通过 bootstrap 脚本获取；README 明确网络前置条件。

## 3. 目标架构与文件边界

### 3.1 运行数据流

```text
scenarios3d/*.json
        |
        v
T1SoccerWorld(MuJoCo model/data) ---> MujocoWorldStateAdapter ---> WorldState
        ^                                                        |
        |                                                        v
T1ActionController <--- RobotAction <--- TeamCoordinator + DecisionFSM
        |
        +--> T1LocomotionController --> joint torques --> mj_step
        +--> T1KickController -------> joint targets --> foot-ball contact
        |
        +--> RunRecorder / 3D metrics / offscreen video / realtime viewer
```

关键原则：3D 模式中，策略读取的状态、窗口显示的状态、日志记录的状态和 evaluator 判定的状态都来自同一个 `MjData`。

### 3.2 CLI 兼容方案

```text
--sim-backend 2d          现有确定性 2D 后端，默认用于测试
--sim-backend mujoco-t1  新 Booster T1 3D 后端

--viewer 2d              俯视实时/离线渲染
--viewer mujoco-preview  当前圆柱 2.5D 窗口
--viewer mujoco          新 T1 3D 实时窗口
--viewer none            无窗口运行
```

原来的 `--viewer mujoco` 在迁移提交中打印一次兼容提示；当新 T1 后端可用后，该名称正式指向 Booster T1 3D。`bash run.sh view` 改为启动 `--sim-backend mujoco-t1 --scenario3d showcase_t1 --viewer mujoco`。

### 3.4 新的内部命令契约

在 `simulation3d/commands.py` 定义：

```python
from dataclasses import dataclass
from enum import Enum

class T1ControlMode(str, Enum):
    DAMP = "damp"
    PREP = "prep"
    WALK = "walk"
    CUSTOM = "custom"
    PROTECT = "protect"

@dataclass(frozen=True)
class BodyVelocityCommand:
    robot_id: int
    vx: float
    vy: float
    yaw_rate: float
    expires_at: float

@dataclass(frozen=True)
class KickCommand:
    robot_id: int
    target_x: float
    target_y: float
    power: float
    requested_at: float
```

速度上限固定从 `configs/3d/soccer_world.json` 读取：`vx`/`vy` 不超过 `0.5 m/s`，`yaw_rate` 不超过 `1.0 rad/s`。过期命令在 WALK 模式输出零速度并保持主动平衡；显式停止按 `WALK -> PREP` 执行，急停按 `任意模式 -> DAMP` 执行。

模式转换必须遵循官方约束：

```text
DAMP -> PREP
PREP -> DAMP | WALK | CUSTOM
WALK -> PREP | DAMP | PROTECT
CUSTOM -> PREP | DAMP | PROTECT
任意故障状态 -> PROTECT
PROTECT --人工/测试重置--> DAMP
```

禁止 `DAMP -> WALK` 和 `WALK -> CUSTOM` 直跳。踢球需要 `WALK -> PREP -> CUSTOM`，踢球结束后 `CUSTOM -> PREP -> WALK`。

## 4. 3D Definition of Done

### 4.1 模型与场景

- [ ] 使用锁定提交中的官方标准版 `T1_23dof.xml`，启动日志输出来源、commit、23 DoF 关节清单和模型变体。
- [ ] `stand_two_t1` 同时加载两台我方 Booster T1，模型、关节、actuator、sensor 名称无冲突。
- [ ] `2v2_attack_defense_t1` 同时加载四台 T1；若性能不满足门槛，允许对手先使用低频 locomotion，但不允许改回圆柱模型。
- [ ] 3D 场景包含足球场、边界、中心线、球门、足球、灯光和至少三个相机视角。
- [ ] 足球、地面、球门和机器人碰撞参数生效，球不会穿透地面或机器人脚部。

### 4.2 控制与状态

- [ ] T1 能从站立姿态稳定保持 10 秒，躯干高度不低于 `0.45 m`，无 NaN。
- [ ] T1 能执行前进、后退、左移、右移、原地旋转和复合行走；每个平移方向净位移至少 `0.4 m`，根节点不是逐帧写入。
- [ ] T1 能转向 `90 deg`，最终偏差不超过 `15 deg`。
- [ ] `WorldState` 从 MuJoCo 读取球、队友、对手、朝向、速度和时间，输出频率至少 30 Hz。
- [ ] DAMP、PREP、WALK、CUSTOM、PROTECT 模式转换符合官方约束，非法转换被拒绝并记录失败码。
- [ ] 控制命令超时后 WALK 输出零速度；机器人跌倒、关节超限或状态非法时进入 PROTECT。
- [ ] 每台 T1 的头部 RGB/depth 相机可输出非空帧；基础协同策略可选择继续使用 MuJoCo 真值状态。
- [ ] WALK 模式通过项目抗扰基准：40 N 侧向力持续 0.1 秒后，2 秒内恢复到躯干高度 `>=0.60 m`、roll/pitch 绝对值 `<=20 deg` 且未跌倒。

### 4.3 协同足球

- [ ] 两台 T1 在 `pass_fixed_t1` 中执行“接近球 -> 对准 -> 踢球 -> 接球者跑位”。
- [ ] 有效传球至少包含一次真实脚球 contact，球沿目标方向移动 `>= 0.8 m`，接球者进入 `0.45 m` 接球区。
- [ ] 传球全程不直接修改球的 `qvel`；调试用 `mock_kick` 运行必须在 summary 中写 `physics_valid=false`。
- [ ] 保留带球、跑位、卡位、射门和 2v2 策略演示；策略失败必须输出原因。

## 8. 风险与停止条件

| 风险 | 可观察信号 | 处理规则 |
| --- | --- | --- |
| 官方模型变更 | body/joint/actuator 名称测试失败 | 保持固定 commit；升级必须单独评审 |
| policy 与 23-DoF XML 不匹配 | 12 维 locomotion 输出无法映射到全身 actuator | 只按名称驱动双腿 12 关节，上身 11 关节交给 posture controller，不扩展或截断 policy |
| 多机器人性能不足 | 两台 T1 实时倍率 `<0.8x` | 先降渲染频率和阴影，不能降低物理精度或改回圆柱模型 |
| T1 站立不稳 | 10 秒内跌倒 | 先修初始 keyframe、PD、接触和 policy；暂停足球策略迁移 |
| 踢球无真实接触 | 球移动但 contact 为 0 | 判定失败；检查是否存在直接 qvel 写入 |
| 踢球动作持续跌倒 | contact 后躯干高度低于门槛 | 调整支撑转移和轨迹；不进入传球验收 |
| 模式转换违反官方约束 | DAMP 直达 WALK、WALK 直达 CUSTOM | 动作拒绝并记录 `INVALID_T1_MODE_TRANSITION`，不得绕过模式机 |
| 3D 与 2D 结果不一致 | 同场景策略事件分歧 | 比较 WorldState 单位、坐标、时间和动作接受语义，不把 2D 结果强行复制到 3D |
| WSLg 窗口失败 | GLFW/OpenGL 初始化错误 | 先用离屏模式跑测试，再修 WSLg；不能宣称实时 3D 展示完成 |
| 外部 ROS2/SDK 不可用 | topic/service 或网络接口缺失 | 不阻塞本地 MuJoCo 3D；真实桥接保持显式未完成 |

以下情况必须停止并报告，不能用 mock 结果冒充完成：

- 官方 T1 模型或 mesh 无法按固定提交获取；
- T1 locomotion policy 无法与所选 XML 对齐；
- 传球只能通过直接修改球速度实现；
- 两台 T1 无法稳定站立或模型命名冲突无法消除；
- 最终视频仍显示圆柱机器人而不是 T1 mesh；
- 自动测试通过，但实时窗口或录像画面为空、严重裁切或没有运动。
