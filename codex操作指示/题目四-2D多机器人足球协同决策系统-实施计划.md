# 题目四：Booster T1 2D 多机器人足球协同决策系统实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 在 WSL2 Ubuntu 22.04 中，以 CPU 可稳定运行的 2D 足球仿真为主环境，实现并自动验收 Booster T1 双机器人传球/接球、带球、跑位、卡位、射门和策略切换闭环，同时保留接入 ROS2、Booster SDK 或其他小组统一接口的适配边界。

**Architecture:** 保留现有 `WorldState -> DecisionFSM -> RobotAction` 主干，在其外补齐可复现的场景加载、团队级协同意图、动作/决策事件、结果判定、指标统计和 2D 录像。仿真器只负责确定性 2D 运动与碰撞，策略层不依赖渲染器、ROS2 或 Booster SDK；真实接口全部放在 `bridge/`，未接通时必须显式报错，不允许用默认假数据伪装成真实运行。

**Tech Stack:** WSL2 + Ubuntu 22.04、Python 3.10、pytest、标准库 dataclasses/json/csv/logging、matplotlib + ffmpeg（仅 2D 录像）、可选 ROS2 Humble 与 Booster Robotics SDK。主线不依赖 CUDA、MuJoCo、Gazebo、Webots 或 3D 人形模型。

## 2. 范围、路线与完成定义

### 2.1 推荐路线

采用“纯 Python 确定性 2D 仿真 + 可选真实接口适配”路线：

```text
JSON 场景
   |
   v
2D Simulator --> WorldState 快照 --> TeamCoordinator --> 每机器人 FSM/策略
      ^                                                   |
      |                                                   v
      +---------------------- RobotAction ----------------+
                               |
                               +--> EventRecorder --> JSONL/CSV/summary.json
                               +--> ScenarioEvaluator --> PASS/FAIL + 指标
                               +--> 2D Renderer --> PNG/MP4（离线、可关闭）
```

选择理由：

- 题目四考核的是协同决策，不是人形动力学或 3D 渲染。
- 6 个 2D 机器人、30 Hz/50 Hz 状态和策略计算可完全使用 CPU，外接 5060 Ti 不影响最低验收。
- 策略只依赖统一接口，后续可把数据源和动作端替换为 ROS2、Booster SDK 或其他组平台。
- 先把结果判定、日志和测试做实，再决定是否接 3D，可显著降低联调风险。

### 2.2 必须完成（Definition of Done）

- [ ] `pass_fixed` 场景中，0 号机器人发起传球，1 号机器人进入 RECEIVE 并在 6 秒内完成接球。
- [ ] 有效传球必须同时满足：出现一次 `PASS_KICKED`；球沿目标方向移动至少 0.8 m；球进入半径 0.35 m 的接球区；接球方与球距离不大于 0.30 m 且球速不高于 1.2 m/s；过程中未出界、未被对手先控制。
- [ ] `dribble_open` 场景中，机器人在 10 秒内让球沿进攻方向净推进至少 1.5 m，且至少 80% 有效帧中机器人与球距离不大于 0.45 m。

- [ ] 决策切换覆盖 `CHASE -> DRIBBLE/PASS -> RECEIVE/SHOOT` 和威胁下的 `BLOCK`；单机器人每 tick 最多一个最终运动动作和一个互斥的踢球动作。

- [ ] 120 秒无窗口仿真无崩溃、无 NaN、实体不越界；`--fast` 模式的实时倍率不低于 20 倍。

- [ ] 生成至少一个非空 2D MP4，画面可辨认球、双方机器人、角色、状态、比分与仿真时间。

## 4. 核心数据契约

### 4.1 坐标和时间约定

- 场地中心为 `(0, 0)`，己方球门在负 X，对方球门在正 X。
- 所有距离单位为米，速度为 m/s，角度为弧度，时间为仿真秒。
- 蓝队 ID 为 `0..N-1`，黄队 ID 为 `10..10+N-1`；真实适配器负责外部 ID 映射。
- `WorldState.timestamp` 必须单调递增；同一 tick 内对所有策略只提供同一份快照。
- 配置中的摩擦使用“每秒衰减率/时间常数”，不得继续使用“每帧乘 0.95”这种与 FPS 绑定的语义。

### 4.2 团队协同意图

在 `decision/team_coordinator.py` 定义：

```python
from dataclasses import dataclass
from enum import Enum
from typing import Optional, Tuple

class PassPhase(str, Enum):
    PLANNED = "planned"
    KICKED = "kicked"
    IN_FLIGHT = "in_flight"
    RECEIVED = "received"
    FAILED = "failed"

@dataclass
class PassIntent:
    passer_id: int
    receiver_id: int
    target: Tuple[float, float]
    created_at: float
    expires_at: float
    phase: PassPhase = PassPhase.PLANNED
    kick_event_id: Optional[str] = None
    failure_code: Optional[str] = None
```

规则：意图存在时锁定传球者与接球者角色；只有 `RECEIVED`、`FAILED` 或超时后才恢复最近球角色分配。

### 4.3 事件类型

在 `common/events.py` 定义三类可 JSON 序列化事件：

```python
from dataclasses import asdict, dataclass, field
from typing import Any, Dict, Optional

@dataclass(frozen=True)
class ActionEvent:
    event_id: str
    tick: int
    timestamp: float
    robot_id: int
    action: str
    params: Dict[str, Any] = field(default_factory=dict)
    accepted: bool = True
    reject_code: Optional[str] = None

@dataclass(frozen=True)
class DecisionEvent:
    tick: int
    timestamp: float
    robot_id: int
    state: str
    role: str
    reason_code: str
    reason: str

@dataclass(frozen=True)
class OutcomeEvent:
    tick: int
    timestamp: float
    outcome: str
    success: bool
    metrics: Dict[str, float] = field(default_factory=dict)
    failure_code: Optional[str] = None
```

统一失败码至少包含：`PASS_BLOCKED`、`PASS_TIMEOUT`、`PASS_OUT_OF_BOUNDS`、`PASS_INTERCEPTED`、`KICK_OUT_OF_RANGE`、`DRIBBLE_LOST`、`STALE_WORLD_STATE`、`INVALID_WORLD_STATE`。

## 6. WSL、外接 5060 Ti 与渲染决策

### 6.1 主线配置

- 决策、2D 仿真、pytest、CSV/JSONL 日志：CPU。
- 无窗口 2D 录像：matplotlib Agg + ffmpeg，CPU。
- 可选实时 2D 窗口：WSLg。
- 不加载 Booster T1 3D 人形网格，不启动多个 3D 人形实例。
- 只有强化学习、视觉神经网络或 3D 动力学扩展需要 CUDA/GPU。

### 6.2 当前 GPU 诊断分支（不阻塞主线）

当前 WSL 内 NVML 初始化失败。确需 GPU 时按以下顺序核查：

Windows PowerShell（管理员）执行：

```powershell
wsl --update
wsl --shutdown
nvidia-smi
```

重新进入 Ubuntu 后执行：

```bash
ls -l /dev/dxg
/usr/lib/wsl/lib/nvidia-smi
```

成功标准：Windows 和 WSL 都能列出外接 RTX 5060 Ti、驱动版本和显存；`/dev/dxg` 存在。若 Windows 侧也失败，先修复显卡驱动、雷电外接显卡连接或权限；不要在 Ubuntu 内安装 Linux NVIDIA 内核驱动。主线 2D 功能继续用 CPU 完成。

## 8. 风险与止损条件

| 风险 | 触发信号 | 处理方式 |
| --- | --- | --- |
| 过早接 3D/实机 | 两天后最低传球测试仍未通过 | 立即回到 2D，冻结 bridge 工作直到核心验收完成 |
| 场景与仿真双状态源 | 日志坐标与画面/动作对象不一致 | 只允许 JSON -> Simulator -> WorldStateProvider 单向链路 |
| 角色震荡 | 相邻 tick 持球者反复交换 | 使用角色最短保持时间、切换差值和 PassIntent 锁定 |
| 重复踢球 | 一个意图出现多个 PASS_KICKED | kick_event_id 幂等；进入 IN_FLIGHT 后禁止再次 kick |
| 测试假阳性 | 只有 tick/无异常断言 | 每个验收测试必须断言 outcome、事件序列和量化指标 |
| WSL `/mnt/c` I/O 慢 | 大量日志导致实时倍率下降 | venv 放 Linux 家目录；轨迹批量缓冲写；必要时把仓库克隆到 `~/projects` |
| 外接 GPU 不可见 | WSL `nvidia-smi` 失败 | 不阻塞 2D；仅在 GPU 扩展阶段修复 Windows 驱动/WSL GPU 通道 |
| 外部接口变化 | 对方字段或 topic 改名 | 只改 bridge 映射和契约 fixture，不改策略层 |
