# 2025-2026 学年《软件工程导论》短学期作业要求
## 基础信息
专业：数据科学与大数据技术
班级：24级一班；24级二班
项目方向：基于Booster T1 机器人的软件工程实践
核心要求：围绕 Booster T1 机器人，采用软件工程思想完成需求分析、系统设计、编码实现、测试验证、文档撰写、版本管理与项目演示。任务最终需形成可运行、可复现、可测试、可展示、可集成的阶段性成果。

## 一、总体要求
### 1. 分组要求
- 每个项目组原则上 8-15 人。
- 每组必须设置项目负责人、需求与文档负责人、架构与接口负责人、功能开发负责人、测试负责人、集成与演示负责人。
- 每名学生必须承担可追踪的独立任务，至少提交 1 个明确模块、1 份阶段记录或 1 份测试/文档成果。
- 禁止单人完成全部工作，也禁止只提交最终成品而无过程记录。
- 允许跨组共享公共接口、样例数据、MuJoCo 环境和通用工具，但各组必须说明本组独立完成内容。

### 2. 项目定位
本短学期作业以“软件工程方法推进复杂机器人任务”为核心。评价重点包括：需求是否清楚、设计是否合理、接口是否规范、开发是否可协作、测试是否充分、系统是否可运行、文档是否完整。

### 3. 技术选型
|方案|推荐技术栈|适用任务|
| ---- | ---- | ---- |
|方案 A：Python 仿真与算法路线|Python + MuJoCo + NumPy/PyTorch + Flask/FastAPI|舞蹈重定向、踢球控制、视觉感知、状态估计|
|方案 B：前后端系统路线|Vue/React + Flask/FastAPI/Django + MySQL/SQLite + WebSocket|任务调度、演示平台、数据管理、可视化看板|
|方案 C：机器人控制路线|C++/Python + MuJoCo + ROS2/自定义接口|动作控制、机器人足球环境、多机器人协作|
|方案 D：混合工程路线|Python + C++ 扩展 + MuJoCo + 前端展示|高性能仿真、视觉处理、强化学习、综合集成|

说明：不强制所有小组使用同一技术栈，但项目二各组必须遵守统一接口规范，避免后期无法集成。

## 二、期末大作业题目及分组
以下题目围绕 Booster T1 机器人任务推进设置。各题目可独立验收，也可作为同一综合项目下的并行子系统共同集成。

### 题目一：Booster T1 舞蹈动作重定向与 MuJoCo 仿真演示系统
1. 项目背景：基于输入舞蹈视频，提取人体动作信息，完成动作数据清洗与机器人动作重定向，使 Booster T1 机器人能够在 MuJoCo 环境中完成舞蹈动作演示。
2. 人数：8-12 人。
3. 核心功能要求：
- 支持指定舞蹈视频输入、读取、抽帧和预处理；
- 完成单人姿态关键点提取，并输出标准化动作数据；
- 完成动作数据插值、平滑、异常点处理和动作片段保存；
- 设计人体骨架到 Booster T1 关节结构的映射关系；
- 生成机器人关节轨迹文件，并在 MuJoCo 中播放；
- 导出机器人舞蹈仿真演示视频和动作数据文件。
4. 软件工程要求：需提交动作处理流程图、模块结构图、MotionData 数据格式说明、重定向接口说明、测试用例和演示视频。
5. 最低验收标准：指定样例视频能够完成“视频输入 - 姿态提取 - 动作清洗 - 动作重定向 - MuJoCo 播放 - 演示视频导出”的完整链路，仿真演示不少于 10 秒。
6. 挑战要求：支持用户自定义上传视频、动作节拍对齐、动作自然性优化、减少机器人抖动或穿模、尝试实机演示。

### 题目二：Booster T1 机器人足球仿真平台与公共接口系统
1. 项目背景：构建统一的 MuJoCo 足球仿真平台，为踢球动作、多机器人协作、定位球调度和视觉感知等小组提供公共环境和接口。
2. 建议人数：6-8 人。
3. 核心功能要求：
- 搭建包含足球场、足球、球门、边界和 Booster T1 机器人的 MuJoCo 场景；
- 设计统一 WorldState 世界状态接口，输出球、机器人、球门和场地信息；
- 设计统一 RobotAction 动作接口，支持站立、移动、转向、踢球、停止和重置；
- 提供 Mock 数据与接口样例，支持其他小组并行开发；
- 提供一键运行脚本、环境配置说明和接口测试脚本；
- 记录系统日志，支持基本的场景复现。
4. 软件工程要求：需提交系统架构图、接口说明文档、WorldState/RobotAction 字段表、接口调用样例、部署说明和联调记录。
5. 最低验收标准：MuJoCo 足球环境能够一键启动，WorldState 能稳定输出球和机器人位置，其他小组能够通过接口调用基础动作或读取样例状态。
6. 挑战要求：支持多机器人加载、日志回放、可视化面板、场景切换、性能监控。

### 题目三：Booster T1 仿人踢球动作控制系统
1. 项目背景：基于 Booster T1 机器人和 MuJoCo 足球环境，实现仿人踢击动作，完成平踢、射门和踢后平衡控制，高踢球作为挑战任务。
2. 建议人数：10-15 人。
3. 核心功能要求：
- 设计踢球动作阶段，包括准备、支撑、摆腿、触球、收腿和恢复；
- 至少实现一种可运行控制方法，如关键帧控制、轨迹规划、PD 控制、模仿学习或强化学习；
- 实现固定球位平踢，使足球朝目标球门方向运动；
- 实现射门效果评估，包括方向、距离、球速、进球情况和踢后稳定性；
- 提供动作参数配置文件，支持调整踢击方向和力度；
- 记录每次实验结果，输出统计表和演示视频。
4. 软件工程要求：需提交动作状态图、控制流程图、算法说明、参数说明、测试脚本、实验记录和失败案例分析。
5. 最低验收标准：固定球位测试中，机器人能够完成平踢动作，使足球朝球门方向运动；至少完成 30 次测试并统计成功率；踢球后系统不能崩溃。
6. 挑战要求：平踢进球成功率达到较高水平、高踢球或挑射、踢后保持平衡、使用强化学习并提供训练曲线。

### 题目四：Booster T1 多机器人足球协同决策系统（我方队伍任务）
1. 项目背景：基于足球场世界状态信息，实现多机器人之间的传球、带球、跑位、卡位和简单攻防决策，体现复杂系统中的模块协作与策略设计。
2. 建议人数：8-12 人。
3. 核心功能要求：
- 基于 WorldState 读取球、队友、对手和球门位置；
- 实现两机器人之间的固定点传球或接球区域判断；
- 实现带球推进，使机器人能够推动足球向指定方向移动；
- 实现简单跑位或卡位策略，根据球和对手位置选择站位；
- 设计传球、带球、射门、卡位之间的策略切换逻辑；
- 提供协作场景演示和策略执行日志。
4. 软件工程要求：需提交协作策略说明、状态机/行为树图、策略接口说明、场景测试报告和多机器人协作演示视频。
5. 最低验收标准：两个机器人能够基于统一 WorldState 完成一次有效传球或一次带球推进，并输出决策过程日志。
6. 挑战要求：完成“传球 - 接球 - 射门”连续流程、实现 2v1 或 2v2 简单攻防、加入对手干扰和策略对比。

### 题目五：Booster T1 外部指令响应与定位球任务调度系统
1. 项目背景：构建机器人足球任务调度系统，使机器人能够根据外部指令完成进场、罚球、角球、边线球等任务流程，重点考查软件工程中的状态管理、任务编排和异常处理。
2. 建议人数：8-12 人。
3. 核心功能要求：
- 设计 TaskCommand 外部指令格式，支持 ENTER_FIELD、PENALTY_KICK、CORNER_KICK、SIDE_KICK、STOP、RESET 等指令；
- 实现指令解析、任务队列和状态切换；
- 使用有限状态机或行为树描述进场、罚球、角球、边线球流程；
- 调用 RobotAction 接口完成移动、站位、瞄准和踢击；
- 支持任务失败、超时、停止、重置等异常处理；
- 记录任务状态、执行耗时、成功与失败原因。
4. 软件工程要求：需提交指令格式说明、状态机图、任务流程图、异常处理说明、日志格式和集成测试报告。
5. 最低验收标准：系统能够接收外部指令，完成进场和罚球两个完整流程，并输出完整状态日志；指令解析正确率应达到 100%。
6. 挑战要求：支持图形界面或网络指令，完成角球和边线球，能与多机器人协作组联动完成后续进攻。

### 题目六：Booster T1 足球视觉感知与世界状态估计系统
1. 项目背景：构建足球场世界状态感知模块，为足球仿真和机器人决策提供球、机器人、球门和场地边界等信息。基础版本可使用 MuJoCo 真值状态，视觉识别和 SLAM 作为进阶方向。
2. 建议人数：10-15 人。
3. 核心功能要求：
- 从 MuJoCo 中读取真值状态，并对外输出 50Hz WorldState；
- 配置仿真相机，采集足球场图像或视频数据；
- 至少完成足球或球门中的一种视觉识别；
- 尝试将图像识别结果转换为世界坐标，并与 MuJoCo 真值进行对比；
- 输出状态来源标记，如 truth、vision、fusion；
- 提供频率、延迟、误差和稳定性测试报告。
4. 软件工程要求：需提交数据采集流程、视觉处理流程图、状态估计接口、误差评估方法、性能测试报告和可视化演示。
5. 最低验收标准：基于 MuJoCo 真值状态稳定输出 50Hz 世界信息，连续运行不少于 2 分钟；完成至少一种视觉识别功能演示。
6. 挑战要求：视觉估计足球位置误差小于 0.5 米，尝试多目标识别、视觉融合、简化 SLAM 或多相机定位。

## 三、统一接口与并行开发要求
### 1. 公共接口优先
项目二所有小组必须围绕 WorldState、RobotAction、TaskCommand 和 EvaluationLog 四类接口协作。第 1 阶段应冻结最小字段，后续只允许兼容性扩展，不允许随意破坏接口。

|接口名称|主要作用|供给方|使用方|
| ---- | ---- | ---- | ---- |
|WorldState|输出球、机器人、球门、场地边界等世界信息|仿真平台组、视觉组|踢球组、协作组、定位球组|
|RobotAction|调用站立、移动、转向、踢球、停止、重置等动作|平台组、踢球组|协作组、定位球组|
|TaskCommand|输入进场、罚球、角球、边线球、停止、重置等指令|定位球组|平台组、协作组、踢球组|
|EvaluationLog|记录执行时间、状态变化、成功率、误差和失败原因|所有小组|测试验收组、教师验收|

### 2. Mock 并行开发
- 公共接口未完成前，各组可使用 Mock 状态数据、预设动作文件、假踢球函数和固定场景进行开发。
- 每组必须提供至少 1 个 Mock 测试样例，保证他组无需等待真实模块即可联调。
- 最终验收时应尽量替换为真实接口；确因难度无法完成时，必须说明替代方案和未完成原因。

## 六、统一验收标准
### 1. 最低通过标准
- 必须有可运行代码或可播放仿真/演示结果；
- 必须有完整 README，能够说明环境配置、启动方法和演示步骤；
- 必须提交需求、设计、测试、分工和总结类文档；
- 必须提供至少一个核心功能演示视频；
- 必须提供测试用例和测试结果，不能只展示成功案例；
- 必须说明每名成员的实际贡献；
- 必须说明未完成内容、原因和后续改进方案。

### 2. 不合格情形
- 无法运行或无法展示任何核心功能；
- 只有概念设计，没有代码、测试或演示；
- 多人小组但分工不清，无法说明个人贡献；
- 未提交核心文档或文档与实际系统严重不符；
- 抄袭他人项目、直接复制网络代码且无说明和修改；
- 演示结果无法复现，且无法说明运行环境和步骤。

### 3. 优秀标准
- 系统结构清晰，接口规范，模块之间能够稳定联调；
- 核心功能不仅能运行，而且有量化测试指标；
- 代码目录规范，注释清楚，配置文件完整，一键运行程度高；
- 能主动记录失败案例并进行分析改进；
- 文档图表规范，能体现软件工程全过程；
- 答辩时能清楚说明技术路线、设计取舍、难点和个人贡献。
