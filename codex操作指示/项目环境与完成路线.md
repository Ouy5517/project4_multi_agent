# Booster T1 多机器人足球协同决策系统：环境准备与完成路线

本文档根据本文件夹中的《软件工程短学期作业要求.docx》、当前仓库代码，以及 Booster 官网开源资料整理。你们当前仓库对应作业题目四：**Booster T1 多机器人足球协同决策系统**。

## 1. 任务目标

课程要求的核心目标不是只写一个能跑的程序，而是用软件工程方法完成一个可协作、可运行、可测试、可展示、可复现的机器人足球决策系统。

题目四的最低验收标准是：

- 两个机器人能够基于统一 `WorldState` 完成一次有效传球，或完成一次带球推进。
- 系统能够输出完整的决策过程日志。
- 需要提交协作策略说明、状态机或行为树图、策略接口说明、场景测试报告和多机器人协作演示视频。

你们当前仓库已经实现了题目四的基础骨架：

- `common/world_state.py`：统一世界状态。
- `common/robot_action.py`：机器人动作接口。
- `strategy/`：传球、带球、射门、跑位、卡位策略。
- `decision/decision_fsm.py`：决策状态机。
- `simulation/field_simulator.py`：自包含 2D mock 足球仿真。
- `tests/`：单元测试和集成测试。
- `main.py`：演示入口，支持 mock 和 real 两种模式。

## 2. Booster 官方资料与你们项目的关系

Booster 官网的开源资料页面给出了几个关键方向：

- Booster Robotics SDK：面向 Booster 机器人二次开发的 SDK，支持 C++ 和 Python。
- ROS2 SDK：提供机器人控制相关的 ROS2 消息和服务定义。
- RoboCup Demo：覆盖感知、定位、决策、执行的 RoboCup 3v3 示例。
- Booster Gym / Booster Train / Booster Deploy / Booster Assets：分别偏强化学习训练、Isaac Lab 训练、部署和模型资源。

对你们题目四来说，最相关的是：

| 官方资料 | 和本项目的关系 |
| --- | --- |
| Booster Robotics SDK | 后续接实机或官方接口时，用于控制机器人或读取状态。 |
| ROS2 SDK | 如果和其他组、仿真平台或真实机器人通过 ROS2 联调，需要使用它的消息和服务定义。 |
| RoboCup Demo | 可以参考其 `vision / brain / game_controller` 分层思路，其中 `brain` 对应你们的策略决策模块。 |
| Booster Assets | 获取 Booster 机器人模型、数据或资源，用于仿真/展示扩展。 |

目前建议不要一开始就把官方 RoboCup Demo 全量并入你们仓库。它是完整比赛级工程，复杂度高。更务实的路线是：先把你们自己的 mock 决策系统做稳定，再逐步把 `WorldState` 和 `RobotAction` 接到 Booster 官方 SDK、ROS2 SDK 或其他小组提供的仿真平台。

- pytest 未安装。

### 3.2 进阶环境：接入 Booster 官方仿真/ROS2/实机时需要

如果你们只做课程题目四的基础验收，当前 Python mock 仿真已经够用。如果要和 Booster 官方资料、MuJoCo、ROS2 或其他小组联调，建议准备 Linux 环境。

推荐方案：

- Windows 电脑保留当前项目开发。
- 安装 WSL2 + Ubuntu 22.04 LTS。
- 在 Ubuntu 中安装 ROS2 Humble、colcon、CMake、gcc/g++、Python 开发工具。
- 在 Ubuntu 中再克隆 Booster 官方 SDK、ROS2 SDK、RoboCup Demo。

原因：

- Booster Robotics SDK 官方预构建环境写的是 Ubuntu 22.04 LTS，CPU 支持 aarch64 和 x86_64，编译器 gcc 11.4.0。

- RoboCup Demo 的构建脚本是 Linux shell 脚本，在 Windows 下容易遇到换行符问题。

下载安装 Git for Windows：

你已经在做这一步。最终目标是让下面命令成功：

```powershell
ssh -T git@github.com
```

成功时通常会看到类似：

```text
Hi 用户名! You've successfully authenticated...
```

注意：

- `.pub` 后缀文件是公钥，可以添加到 GitHub。
- 没有 `.pub` 后缀的是私钥，不能发给任何人，也不能提交进仓库。
- 当前仓库远程地址应为：

```powershell
git remote -v
```

应看到：

```text
origin  git@github.com:Ouy5517/project4_multi_agent.git (fetch)
origin  git@github.com:Ouy5517/project4_multi_agent.git (push)
```

### 4.4 解决 Windows 中文/emoji 输出问题

当前 `main.py` 在 Windows 默认 GBK 控制台下可能因为 emoji 输出报错。每次运行前可以设置：

目标：

- 明确 `WorldState` 里必须包含哪些字段。
- 明确 `RobotAction` 能提供哪些动作。

建议最小字段：

```text

  goals.yellow

  field.height

  move_to(robot_id, target)
  turn_to(robot_id, angle)
  dribble(robot_id, direction, speed)
  kick(robot_id, target, power)
  stop(robot_id)
  reset(robot_id)
```

产出：

- `docs/API.md`
- `docs/WorldState字段表.md`
- `docs/RobotAction接口说明.md`
- 至少 1 个 mock 输入样例。

### 第 3 步：完成最低验收功能

目标：

- 稳定完成一次有效传球，或一次带球推进。
- 输出可读的决策日志。

建议优先完成传球闭环：

1. `WorldState` 读取球、己方机器人、对手、球门位置。
2. 判断当前最近球的机器人为持球者。
3. 找到接球队友。
4. 判断接球区域是否安全。
5. 执行 `PASS`。
6. 接球队友移动到接球点。
7. 日志记录 `CHASE -> PASS -> RECEIVE` 或类似状态变化。

验收标准：

- 日志中能看到两个机器人参与。
- 能看到传球开始、传球目标、传球结果。
- 如果失败，要能说明失败原因，例如距离过远、对手拦截、角度不合适。

产出：

- 传球演示命令。
- 传球演示视频。
- 传球场景测试报告。

### 第 4 步：补齐带球、跑位、卡位和射门

目标：

- 让系统不只是单点传球，而是体现策略切换。

建议策略优先级：

```text
如果球在己方控制范围内：
  如果射门条件好 -> SHOOT
  如果队友位置更好 -> PASS
  否则 -> DRIBBLE
如果球不在控制范围内：
  最近机器人 -> CHASE
  其他机器人 -> POSITION
如果对手威胁球门：
  防守机器人 -> BLOCK
```

每个策略都应有：

- 输入：`WorldState`
- 输出：`RobotAction`
- 日志：策略选择原因
- 测试：至少 2-3 条用例

### 第 5 步：完成测试体系

## 9. 如何结合 Booster 官方开源项目继续扩展

### 9.1 参考 RoboCup Demo 的系统分层

Booster RoboCup Demo 官方说明中包含三个程序：

- `vision`：检测机器人、足球和场地，并计算坐标。
- `brain`：读取视觉数据和比赛控制数据，做判断并控制机器人。
- `game_controller`：读取裁判机广播数据，转换成 ROS2 topic。

你们项目可以对应为：

```text
vision / 仿真平台 / 其他小组 WorldState -> common/world_state.py
brain -> decision/ + strategy/
robot control -> common/robot_action.py + bridge/
logs -> outputs/
```

### 9.2 接 ROS2 SDK 的方向

ROS2 SDK 提供机器人控制相关的消息和服务定义，例如底层状态、IMU、低层命令、电机状态、RPC 服务等。你们不需要一开始就使用全部消息，只需要把它看作未来真实接口来源。

建议做一个适配层：

```text
bridge/
  ros2_world_state_adapter.py
  ros2_robot_action_adapter.py
```

适配层职责：

- 把 ROS2 topic/service 转换成你们统一的 `WorldState`。
- 把你们的 `RobotAction` 转换成 ROS2 控制命令。
- 保持 `strategy/` 和 `decision/` 不依赖 ROS2。

这样做的好处是：即使暂时没有真实机器人，策略代码仍然能在 mock 仿真里测试。

### 9.3 接 Booster Robotics SDK 的方向

Booster Robotics SDK 支持 C++ 和 Python。官方推荐的 Python 使用方式是安装预构建包：

```bash
pip install booster_robotics_sdk_python --user
```

如果后续要接真实机器人或官方 SDK，可以在 `bridge/real_robot_action.py` 中实现：

```text
RobotAction.move_to -> SDK 移动接口
RobotAction.turn_to -> SDK 转向接口
RobotAction.kick -> SDK 踢球/动作接口
RobotAction.stop -> SDK 停止接口
```

课程阶段不建议把真实 SDK 调用写死在策略里，应统一放在 `bridge/`。

## 11. 风险与规避

| 风险 | 表现 | 规避办法 |
| --- | --- | --- |
| Git 权限没配好 | 不能 pull/push | 先确保 `ssh -T git@github.com` 成功 |
| 依赖缺失 | `No module named pytest` | 安装 `pytest matplotlib` |
| Windows 编码问题 | emoji 输出报错 | 设置 `$env:PYTHONUTF8="1"` |
| 公共接口频繁变动 | 各模块无法集成 | 第 2 阶段冻结最小字段，只做兼容扩展 |
| 只演示成功案例 | 测试报告不合格 | 加入失败案例和异常处理说明 |
| 直接复制官方 Demo | 解释不清楚个人贡献 | 只参考架构，核心策略和文档自己实现 |
| 没有过程记录 | 分工不可追踪 | 每人用 Git 分支、提交记录、会议记录证明贡献 |
